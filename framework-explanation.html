Key principle: Tests never contain selectors or URLs. Those live in page objects and the ENV class respectively. This means a UI change requires updating only the page object, not every test that uses it.

2. Page Object Model (POM) Architecture

The Page Object Model is a design pattern where every screen (or section of a screen) has its own class. Tests talk to page objects; page objects talk to the browser. This gives you a single place to update when the UI changes.

Class Hierarchy

BasePage
  └── LoginPage
  └── ProfilePage
  └── TimeOffPage

BasePage holds utilities every page needs: goto(url), pause(ms), and a helper to click a button until it disappears. All page classes extend it, so they all inherit these methods automatically.

Locators as Getters (not stored in the constructor)

Every selector is defined as a TypeScript get property, not a variable set in the constructor. This matters because Playwright locators are lazy — they do not interact with the DOM until an action is performed. Re-creating the locator on each access avoids stale element references:

// CORRECT — fresh locator every time
get firstNameInput(): Locator {
  return this.page.locator('input[name="firstName"]');
}

// AVOID — if the DOM changes after page load, this reference goes stale
constructor(page: Page) {
  this.firstNameInput = page.locator('input[name="firstName"]'); // fragile
}

Actions as Methods

Multi-step interactions are encapsulated as named methods. Tests call a descriptive method, not a chain of clicks:

// In ProfilePage
async updateName(firstName: string, lastName: string): Promise<void> {
  await this.firstNameInput.clear();
  await this.firstNameInput.fill(firstName);
  await this.lastNameInput.clear();
  await this.lastNameInput.fill(lastName);
  await this.updateButton.click();
}

// In the test — clean and readable
await profilePage.updateName('AutoTest', 'User');

3. Key Files Explained

playwright.config.ts

The central configuration file that controls how all tests run. The most important settings:

SettingValueWhy
globalSetup./src/utils/globalSetupRuns before any test — loads the .env file into process.env
storageStateauthentication.jsonDefault session for all tests (authenticated)
dependencies: ['setup']auth.setup projectForces auth.setup.ts to run before any spec file
viewport1920×1080Consistent full-size window — avoids mobile layout issues
baseURLENV.BASE_URLCentralises the app URL so tests don't hardcode it

The config defines two Playwright "projects": setup (runs auth.setup.ts once) and Google Chrome (runs all spec files, depends on setup finishing first).

auth.setup.ts

This is the login-once file. It performs a real browser login, waits for the dashboard URL to confirm success, then saves the entire browser session (cookies, localStorage, sessionStorage) to authentication.json. Every subsequent test loads that file instead of logging in again.

const authFile = 'src/utils/authentication.json';

setup('save authenticated session', async ({ page }) => {
  await page.goto(ENV.LOGIN_LINK);
  await page.locator('input[name="email"]').fill(ENV.VALID_EMAIL);
  await page.locator('input[name="password"]').fill(ENV.VALID_PASSWORD);
  await page.getByRole('button', { name: 'Sign in' }).click();
  await expect(page).toHaveURL(ENV.HOME_PAGE_LINK, { timeout: 15000 });
  await page.context().storageState({ path: authFile }); // ← saves session
});

globalSetup.ts

Runs once before the entire test suite (not before each test). Its only job is to read the correct .env file based on the test_env environment variable and push its values into process.env:

import * as dotenv from 'dotenv';

async function globalSetup() {
  if (process.env.test_env) {
    dotenv.config({
      path: `helper/env/.env.${process.env.test_env}`,
      override: true,
    });
  }
}

export default globalSetup;

Running test_env=test npx playwright test causes this to load .env.test. Changing to test_env=stage loads .env.stage — no test code changes required.

env.ts

A typed class that wraps process.env. Without this, TypeScript would have no idea what keys exist in process.env and would throw type errors. Every env variable used in tests is declared here as a public static property:

export default class ENV {
  public static BASE_URL = process.env.BASE_URL;
  public static LOGIN_LINK = process.env.LOGIN_LINK;
  public static HOME_PAGE_LINK = process.env.HOME_PAGE_LINK;
  public static TIME_OFF_URL = process.env.TIME_OFF_URL;
  public static VALID_EMAIL = process.env.VALID_EMAIL;
  public static VALID_PASSWORD = process.env.VALID_PASSWORD;
  public static INVALID_CREDENTIALS_MESSAGE = process.env.INVALID_CREDENTIALS_MESSAGE;
  public static NONEXISTING_EMAIL = process.env.NONEXISTING_EMAIL;
  public static PROFILE_REQUIRED_ERROR = process.env.PROFILE_REQUIRED_ERROR;
  public static PROFILE_SUCCESS_MESSAGE = process.env.PROFILE_SUCCESS_MESSAGE;
}

4. End-to-End Flow (How a Test Runs)

When you run npm run test:chrome, here is the exact sequence of events:

  1. globalSetup.ts fires. It reads test_env from the shell and loads the matching .env file. All process.env values are now populated.
  2. auth.setup.ts fires (setup project). A fresh browser opens, logs in, waits for the dashboard URL, saves the session to authentication.json, then closes.
  3. Google Chrome project starts. Each spec file runs in its own browser context. That context is pre-loaded with the session from authentication.json, so every test starts already logged in.
  4. Each test's beforeEach runs. For most tests, this navigates to the home page. For login tests, the test.use({ storageState: { cookies: [], origins: [] } }) override wipes the session first so the test sees a logged-out state.
  5. The test body runs. It calls page object methods. The page object locators interact with the live DOM. Assertions use expect() with built-in retry/wait logic.
  6. After all tests complete, Playwright generates the HTML report (stored in playwright-report/).

6. How to Add New Tests

Follow these four steps every time you add coverage for a new feature:

Step 1 — Add selectors to the relevant page object (or create a new one)

If the feature is on an existing page, add a getter to that page's class. If it's a new page, create a new src/pages/featurePage.ts that extends BasePage.

// In src/pages/featurePage.ts
import { Page, Locator } from '@playwright/test';
import { BasePage } from './basePage';

export class FeaturePage extends BasePage {
  constructor(page: Page) { super(page); }

  get someButton(): Locator {
    return this.page.locator('button.some-class');
  }

  async doSomething(): Promise<void> {
    await this.someButton.click();
  }
}

Step 2 — Add any new environment variables to .env.test and env.ts

If your test asserts a specific message or navigates to a specific URL, add it to the env file first:

# In helper/env/.env.test
FEATURE_SUCCESS_MSG = "Action completed!"
FEATURE_URL = "https://qa.rooster-next-level.click/feature"
// In src/utils/env.ts
public static FEATURE_SUCCESS_MSG = process.env.FEATURE_SUCCESS_MSG;
public static FEATURE_URL = process.env.FEATURE_URL;

Step 3 — Write the test in a spec file

Either add to an existing spec or create a new one:

// src/tests/feature.spec.ts
import { test, expect } from '@playwright/test';
import ENV from '../utils/env';
import { FeaturePage } from '../pages/featurePage';

test.describe.serial('Feature tests', () => {
  let featurePage: FeaturePage;

  test.beforeEach(async ({ page }) => {
    featurePage = new FeaturePage(page);
    await page.goto(ENV.HOME_PAGE_LINK);
  });

  test('Should do the thing', async ({ page }) => {
    await test.step('Navigate to feature', async () => {
      await featurePage.doSomething();
      await expect(page).toHaveURL(ENV.FEATURE_URL, { timeout: 10000 });
    });

    await test.step('Verify success message', async () => {
      // assertions here
    });
  });
});

Step 4 — Run your test to verify

npm run test:chrome -- --grep "Should do the thing"
Checklist before committing a new test:
✓ Selectors are in the page object, not in the test
✓ Asserted strings are in .env.test and referenced via ENV class
✓ Test uses test.step() to group logical steps
✓ Test is inside a test.describe.serial() block
✓ Test passes 3 consecutive times without flakiness

7. Common Pitfalls & How to Avoid Them

PitfallSymptomFix
Strict mode violation Test throws "locator resolved to X elements" error Add .first(), .nth(n), or use a more specific selector. Never use a selector that matches multiple elements when you intend to act on one.
Login running on every test auth.setup.ts logs in, then each test logs in again — slow and redundant Ensure playwright.config.ts sets storageState: 'src/utils/authentication.json' on the Google Chrome project and that auth.setup.ts is listed as a dependency.
Logout breaking subsequent tests Tests after the logout test fail with "not authenticated" errors Always put the logout test LAST in a serial block. Logout invalidates the server session — tests that run after it will have stale cookies.
Hardcoded URLs in tests Tests break when the environment changes; hard to maintain Always use ENV.SOME_URL. The URL lives in .env.test. Changing environments means only changing the env file.
Selectors in test files A single UI change forces updates across many test files All selectors belong in page object getters. Tests only call page object methods.
Missing await Test passes unreliably or assertions fire before actions complete Every Playwright action and assertion must be awaited. TypeScript will warn about this if you have strict mode on.
Using page.waitForTimeout() (fixed sleeps) Tests are slow and still flaky on slow networks Use Playwright's built-in auto-wait: toBeVisible({ timeout: 5000 }), toHaveURL(), waitFor({ state: 'visible' }). These retry until the condition is met.
ENV variable reads as undefined Assertion fails comparing against undefined; test passes for wrong reason Check that: (1) the key exists in .env.test, (2) it is declared in env.ts, (3) test_env is being passed correctly when running the test command.
Test state leak between tests Passing tests depend on a previous test's side effects; tests fail when run alone Each test must set up its own preconditions in beforeEach. Never rely on state left by a previous test unless the entire file is serial and the dependency is intentional and documented.
